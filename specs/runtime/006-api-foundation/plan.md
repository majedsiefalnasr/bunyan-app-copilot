## Summary

Establish the complete API foundation for the Bunyan platform by formalizing versioned route organization, adding a `BaseApiController` in the `Api\V1\` namespace, creating a `BaseApiResource` with pagination support, registering three new named rate limiters, creating the missing `config/cors.php`, adding a health-check endpoint at `GET /api/health`, and scaffolding OpenAPI 3.0 documentation via `darkaonline/l5-swagger`.

All work extends the existing infrastructure (Kernel.php global middleware, AppServiceProvider rate limiters, ApiResponseTrait) — no existing middleware or response contracts are modified.

## Phase 1: Base Classes

### Step 1.1 — `BaseApiController`

**File:** `app/Http/Controllers/Api/V1/BaseApiController.php`
**Namespace:** `App\Http\Controllers\Api\V1`
**Extends:** `App\Http\Controllers\Api\BaseController`

**Responsibility:**

- Inherits `success()` and `error()` from `BaseController` (via `ApiResponseTrait`)
- Adds `paginated()` as a new protected method (not in trait — see R-05)
- Provides PHPDoc class header with `@OA\Tag` (to be referenced by child controllers)

**`paginated()` signature and response shape:**

```php
/**
 * @param  \Illuminate\Contracts\Pagination\LengthAwarePaginator|\Illuminate\Http\Resources\Json\ResourceCollection  $collection
 * @param  int  $statusCode
 * @return \Illuminate\Http\JsonResponse
 */
protected function paginated(
    mixed $collection,
    int $statusCode = 200,
): JsonResponse
```

Response shape (FR-010):

```json
{
  "success": true,
  "data": [...],
  "meta": {
    "current_page": 1,
    "per_page": 15,
    "total": 100,
    "last_page": 7
  },
  "error": null
}
```

The `meta` is extracted from the paginator's `->currentPage()`, `->perPage()`, `->total()`, `->lastPage()`. If `$collection` is a `ResourceCollection` backed by a `LengthAwarePaginator`, extract the paginator from `$collection->resource`.

**Important:** `BaseApiController` must NOT duplicate `success()` or `error()`. They are inherited unchanged.

### Step 1.2 — `BaseApiResource`

**File:** `app/Http/Resources/BaseApiResource.php`
**Namespace:** `App\Http\Resources`
**Extends:** `Illuminate\Http\Resources\Json\JsonResource`
**Declaration:** `abstract class BaseApiResource` (MUST use `abstract` keyword — class contains `abstract toArray()` and PHP will not compile a non-abstract class with abstract methods)

**Responsibilities:**

- Sets `public static $wrap = 'data'` (FR-013)
- Abstract `toArray()` — child classes implement their own field mapping
- Static `paginatedCollection()` helper:

```php
public static function paginatedCollection(
    \Illuminate\Contracts\Pagination\LengthAwarePaginator $paginator,
): array
```

Returns an array compatible with `paginated()` in `BaseApiController` — delegates pagination meta extraction to a shared location.

- `withResponse()` override: adds `X-Correlation-ID` to the response headers when the correlation ID is present on the request (consistent with `ApiResponseTrait`).

**Note on `$wrap`:** Setting `$wrap = 'data'` means Laravel will automatically wrap single resource responses in `{"data": {...}}`. For collection responses managed via `BaseApiController::paginated()`, the wrapping is handled manually to include the `meta` key. This avoids double-wrapping.

### Step 1.3 — Migrate Existing Resources (FR-015)

Change the `extends` declaration in:

| File                     | Old            | New               |
| ------------------------ | -------------- | ----------------- |
| `UserResource.php`       | `JsonResource` | `BaseApiResource` |
| `RoleResource.php`       | `JsonResource` | `BaseApiResource` |
| `PermissionResource.php` | `JsonResource` | `BaseApiResource` |
| `UserRoleResource.php`   | `JsonResource` | `BaseApiResource` |

**Risk:** `$wrap = 'data'` changes response envelope. Existing tests may need updating if they assert `response['email']` directly rather than `response['data']['email']`. **Audit all existing resource tests before migrating.** If tests assert the unwrapped shape, add `->withoutWrapping()` in test setup or update assertions.

**Mitigation:** Run full test suite after each resource migration. Fix one at a time.

## Phase 3: Rate Limiters

**File:** `app/Providers/AppServiceProvider.php` — append to existing `RateLimiter::for()` block in `boot()`.

### New Limiters to Add

```php
RateLimiter::for('api-authenticated', function (Request $request) {
    // 60 req/min per authenticated user; fall back to IP for unauthenticated requests
    $key = $request->user()?->id
        ? 'user:'.$request->user()->id
        : 'ip:'.$request->ip();

    return Limit::perMinute(60)->by($key)->response(function () {
        return app(Handler::class)->render(
            app('request'),
            new \Symfony\Component\HttpKernel\Exception\TooManyRequestsHttpException()
        );
    });
});

RateLimiter::for('api-public', function (Request $request) {
    // 10 req/min per IP for unauthenticated/public routes
    return Limit::perMinute(10)->by($request->ip());
});

RateLimiter::for('api-admin', function (Request $request) {
    // 300 req/min per authenticated admin user — parentheses enforce correct null-coalescing
    $key = 'admin:'.($request->user()?->id ?? $request->ip());
    return Limit::perMinute(300)->by($key);
});
```

**Note on FR-025 (429 error contract):** Laravel's default ThrottleRequests middleware returns a plain 429. To return the unified error contract body (`error.code = "RATE_LIMIT_EXCEEDED"`), the Exception Handler must be extended. Check `app/Exceptions/Handler.php` — if it already handles `ThrottleRequestsException` (from prior stages), verify it returns the contract body. If not, add a handler in Phase 3.

**REQUIRED: Retry-After headers on 429 responses (Architecture Guardian — BLOCKED item):**
Update `handleRateLimitException` in `app/Exceptions/Handler.php` to extract and forward rate limit headers from the `TooManyRequestsHttpException`. The current implementation discards these headers when constructing the JSON response:

```php
private function handleRateLimitException(Throwable $e): JsonResponse
{
    // Extract headers from the original exception (Retry-After, X-RateLimit-Limit, X-RateLimit-Remaining)
    $headers = method_exists($e, 'getHeaders') ? $e->getHeaders() : [];
    return $this->errorResponse(
        message: __('errors.rate_limit_exceeded'),
        code: ApiErrorCode::RATE_LIMIT_EXCEEDED,
        statusCode: 429,
        headers: $headers,  // Forward Retry-After + X-RateLimit-* headers
    );
}
```

This ensures `US3 AC4` is satisfied: `Retry-After`, `X-RateLimit-Limit`, and `X-RateLimit-Remaining` are present on all 429 responses.

**Rate limit Redis keys (documented in `data-model.md`):**

- `api-authenticated`: `laravel_cache:limiter:user:{userId}` (or `ip:{ip}` fallback)
- `api-public`: `laravel_cache:limiter:ip:{ip}`
- `api-admin`: `laravel_cache:limiter:admin:{userId}`

## Phase 4: Health Check

### Step 4.1 — `HealthController`

**File:** `app/Http/Controllers/Api/HealthController.php`
**Namespace:** `App\Http\Controllers\Api`
**Note on 503 health check error contract:** When DB is unreachable, HTTP 503 is returned with `success: false, data: null, error: { code: "HEALTH_CHECK_FAILED", message: ..., details: { status: "unhealthy", checks: {...probe statuses}, version, environment, timestamp } }`. Probe data lives in `error.details`, not in `data` — this conforms to the AGENTS.md unified error contract requiring `data: null` on all error responses. `SERVER_ERROR` is NOT used here — `HEALTH_CHECK_FAILED` is the machine-readable signal for a deliberate, handled probe failure.

```php
namespace App\Http\Controllers\Api;

use Illuminate\Routing\Controller;
use Illuminate\Support\Facades\Cache;
use Illuminate\Support\Facades\DB;

class HealthController extends Controller
{
    public function check(): \Illuminate\Http\JsonResponse
    {
        $checks = $this->runChecks();
        $allHealthy = $checks['database'] && $checks['cache'];
        $status = $allHealthy ? 'healthy' : ($checks['database'] ? 'degraded' : 'unhealthy');
        $httpStatus = $checks['database'] ? 200 : 503;
        $timestamp = now()->toIso8601String();

        if ($httpStatus === 503) {
            // Error contract: data is null on error responses
            // Probe details go into error.details for monitoring tool consumption
            return response()->json([
                'success' => false,
                'data'    => null,
                'error'   => [
                    'code'    => 'HEALTH_CHECK_FAILED',
                    'message' => __('errors.health_check_failed', [], 'en'),
                    'details' => [
                        'status'      => 'unhealthy',
                        'version'     => config('app.version', '1.0.0'),
                        'environment' => app()->environment(),
                        'checks'      => $checks,
                        'timestamp'   => $timestamp,
                    ],
                ],
            ], 503);
        }

        return response()->json([
            'success' => true,
            'data'    => [
                'status'      => $status,
                'version'     => config('app.version', '1.0.0'),
                'environment' => app()->environment(),
                'checks'      => $checks,
                'timestamp'   => $timestamp,
            ],
            'error'   => null,
        ], 200);
    }

    private function runChecks(): array
    {
        return [
            'database' => $this->checkDatabase(),
            'cache'    => $this->checkCache(),
        ];
    }

    private function checkDatabase(): bool
    {
        try {
            DB::select('SELECT 1');
            return true;
        } catch (\Throwable) {
            return false;
        }
    }

    private function checkCache(): bool
    {
        try {
            $key = '_health_probe_'.uniqid();
            Cache::put($key, 1, 5);
            $result = Cache::get($key);
            Cache::forget($key);
            return $result === 1;
        } catch (\Throwable) {
            return false;
        }
    }
}
```

**Status logic (FR-039):**

- DB ✅, Cache ✅ → HTTP 200, `status = "healthy"`
- DB ✅, Cache ❌ → HTTP 200, `status = "degraded"` (non-fatal)
- DB ❌ (any) → HTTP 503, `status = "unhealthy"`

**Logging exclusion (FR-040):** `RequestResponseLoggingMiddleware` already has a skip path list. Verify `health` is included. If not, add it (no Kernel.php change needed — only the middleware class itself).

```php
public function boot(): void
{
    // ...

    // Guard: wildcard CORS with credentials is rejected by browsers and insecure
    if (config('cors.supports_credentials') && in_array('*', (array) config('cors.allowed_origins'), true)) {
        if (!app()->isLocal()) {
            throw new \InvalidArgumentException(
```

## Decisions That Differ from Original Spec Anticipation

1. **No Kernel.php changes (CLR-01 alignment):** FR-016–FR-018 in the spec anticipate Kernel.php middleware changes. Research confirmed they are already fully satisfied. Zero changes to Kernel.php.

2. **`paginated()` on `BaseApiController`, not `ApiResponseTrait`:** The spec implies `paginated()` is inherited (FR-007 says "or directly use `ApiResponseTrait`"). Since `ApiResponseTrait` currently lacks `paginated()` and adding it risks side-effects on all trait users, it is added directly to `BaseApiController` as a protected method.

3. **No `config/cors.php` migration from vendor:publish:** Laravel 13 does not ship cors.php in the default scaffold. Instead of running `vendor:publish --tag=laravel-cors` (which publishes a framework template), the file is created directly with project-specific values. This avoids overwriting customizations if the command is run again.

4. **`HealthController` does not extend `BaseController`:** CLR-04 exempts it from the service/repository pattern. Since `BaseController` only provides `ApiResponseTrait`, and `HealthController` constructs its response manually (to support HTTP 503 with the partially-compliant error body), using `response()->json()` directly is consistent and avoids trait method constraints. If later unified, `ApiResponseTrait::success()` can be added — but that's future work.

5. **Route sub-file path is `routes/api/v1/*.php` not `routes/api/*.php`:** The spec says `routes/api/auth.php` (flat) but given the route prefix is `v1`, nesting under `routes/api/v1/` makes the version explicit in the filesystem and supports future `v2` addition without collision.
